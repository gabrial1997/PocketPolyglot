# v0.1.2 — "Round 1: see == hear" fidelity pass

> Design doc / spec. Status: **approved 2026-06-18**, ready for an implementation plan.
> Authored from a brainstorming session on three founder-reported issues ("round 1").
> Read `CLAUDE.md` (the non-negotiable card boundary + locked product constraints) and
> `handoverv0.1.0.md` (current build state) alongside this.

## Goal

Every card plays the **right** clip for what it shows. The word the learner **sees** is always
the same word they **hear** — the *see == hear invariant*. Fix the three reported screens **and**
the root cause of each, so the whole drill / diphthong / say-it family is correct, not just the
three screens in the bug reports.

This is the **correctness/fidelity** framing the founder chose — not a minimal one-line patch.
Ships as v0.1.2: **app code + re-seeded Supabase content together**, verified end-to-end on a
real iPhone in light + dark.

### The invariant (the spine of this work)

> **On any perception or pronunciation card, the word displayed and the word played are
> identical.** Translation-recall cards (e.g. the phrase "say it" card, which cues in English and
> asks the learner to *produce* the Latvian from memory) are the one deliberate exception — there,
> seeing English while producing/hearing Latvian *is* the exercise.

## Background — why these are real bugs

Confirmed at the code level + by inspecting the live DB and downloading the audio objects:

- **Issue 1 — `ie` glide plays "lieta".** The diphthong drill's data carries only the
  contrast-word clips (`drill-lieta.mp3` = the whole word "lieta"). There is **no isolated-glide
  audio anywhere** in the pipeline. The "Meet the glide" intro step (`DiphthongDrillScreen`) plays
  `item.audio.nativeUrl` = the whole word. The card meant to teach the glide in isolation has
  nothing isolated to play.
- **Issue 2 — drill "say it back" shows `lācis`, plays a different word.** `DrillScreen` hardcodes
  the say-it word to `item.target` = `row.a` ("lācis"), but the play orb plays
  `item.audio.nativeUrl` = the **stimulus** clip (`drill-loti.mp3`). So the screen shows the
  word that was *not* played, and plays the stimulus (the correct answer's word). Two unrelated
  words on one screen → violates see == hear. Additionally, the per-option clips
  (`drill-loti-a` = "lācis", `drill-loti-b` = "ļoti") are uploaded to Storage by the seeder but
  **never written to the DB and never mapped into the `ReviewItem`**, so no per-side playback is
  even possible. (The six drill clips were downloaded and are all distinct files — this is not a
  "two rows share one file" bug; the founder reported hearing "lieta", which `drill-loti.mp3` is
  not, so a clip-content verification is included below.)
- **Issue 3 — phrase card not centered + say-it cue.** `CardShell` sets `justifyContent: 'center'`
  (vertical) but **no `alignItems`/`textAlign` center**, so every CardShell-based card renders
  **left-aligned** — play orb and mic hug the left edge with dead space below. The mockups
  (`mockups/phrase_cards.png`) are fully centered. Separately, `PhraseSayIt` shows only the
  English gloss with a play orb that lets the learner hear the answer before recording (a recall
  leak).

## Non-negotiable constraints (must hold through every change)

From `CLAUDE.md`:

- **Cards are pure: data-in (`ReviewItem`) / events-out (`CardResult`).** Fixes land as item
  fields + theme tokens + card-local UI state + at most one new mapper field. No card imports a
  service; no hardcoded content.
- **Theme tokens only; light + dark both verified** for every visual change.
- **`CardKind` `id`/`k` strings stay stable.**
- **Wrong-answer rule untouched** (red "Not quite — give it another try.", no reveal, no advance).
- **No gamification.** The unlock chime stays the only celebratory beat.
- Each **live migration needs its own explicit user authorization** (the permission classifier
  gates them individually).

## Design

### Issue 1 — isolated `ie` glide (recorded, stubbed now)

**Data model.** Add `glide_audio_url text` to `minimal_pairs` (migration **`0007`**). Populated
only for diphthong drills; `null` for plain consonant pairs.

**Content.** The "Meet the glide" step needs a clip of the **isolated glide**, not the word.
Decision: the real clip will be **recorded** (the first real recording in the app — candidate for
Elizabete's voice). For v0.1.2 we ship a **placeholder** glide clip at a known asset path
(`content-pipeline/assets/glide-ie.mp3`), clearly marked temporary. The plan picks the simplest
stub available in the build environment — preferred: a short "ie" segment trimmed from the
existing `drill-lieta` render (most faithful); acceptable fallback: a brief TTS of the glide. The
stub exists to make the wiring verifiable on the phone, not to be the final sound.
**Swap path:** drop the real recording at that asset path and re-run the seeder — **no code
change**.

**App.** Mapper carries `glide.audioUrl` (extend the existing `ReviewGlide` shape /
`item.glide`). `DiphthongDrillScreen`'s "Meet the glide" intro step plays `item.glide.audioUrl`
when present (falling back to the word clip only if it's missing). The later "hear it in a word"
and contrast steps continue to play the word clips. The on-screen "ie" hero + "tap to hear the
glide" copy already match the mockup — only the *audio source* for that one tap changes.

### Issue 2 — drill say-it-back: see == hear

**App (the core fix).** In `DrillScreen`, the say-it-back word **is the stimulus word** — the word
that was played and that the learner correctly identified — **shown and played identically**:

- The say-it label = the stimulus word (the correct side's word), not `row.a`.
- The play orb on the say-it step plays the stimulus clip (same word).
- Result: see == hear by construction, and the pedagogy is corrected (today it asks the learner
  to say back the word that was *not* played).

**Root-cause close — per-option audio.** Extend the pair data so each side has its own clip:

- Add `a_audio_url`, `b_audio_url` to `minimal_pairs` (part of migration **`0007`**), populated
  from the already-uploaded `<slug>-a` / `<slug>-b` Storage objects (seeder change: write the URLs
  it currently discards).
- Map them into `item.pair` (`aAudioUrl`, `bAudioUrl`).
- Any per-side playback then plays the correct word. (Tapping a choice button currently plays
  nothing; this design keeps choice-buttons silent unless a later UX decision adds per-side
  playback — but the *data* is now correct and available, closing the root gap.)

**Verification task (not an assumption).** Transcribe/verify `golden/drill-loti.mp3` actually
says "ļoti". The founder reported hearing "lieta" on the `lācis` screen; the code path plays
`drill-loti.mp3`. If the stored clip's content is wrong (bad TTS input), regenerate it. The
see == hear fix is required regardless of the outcome.

### Issue 3 — phrase card layout + say-it cue

**Layout.** Fix `CardShell` to center horizontally: add `alignItems: 'center'` to the body and
`textAlign: 'center'` to the text styles, matching every mockup. This intentionally affects **all**
CardShell-based cards (they are all meant to be centered). Snapshots will change — review and
update them intentionally (`jest -u`), do not blind-accept.

**Phrase say-it cue (founder's choice: English-only, no pre-hear).** Rework `PhraseSayIt`:

- **Cue stage:** show the **English** prompt only; **no play orb** (no hearing the answer before
  producing it). A single mic to record.
- **After recording:** show the **compare** — *hear native* vs *hear you* — wiring `onPlayCompare`
  (which already exists in `cardWiring` but `PhraseSayIt` does not currently use). Then the
  self-rate Good/Again.
- This is true productive recall and keeps the see == hear exception explicit and intentional.

### Content / delivery

Two delivery channels, coordinated in one cycle:

- **App code** (CardShell, mappers, `DrillScreen`, `DiphthongDrillScreen`, `PhraseSayIt`,
  types) → ships in the JS bundle / app build.
- **Content** (the placeholder glide clip; `glide_audio_url` + `a_audio_url`/`b_audio_url`
  columns and values; any regenerated drill clip) → ships via Supabase: migration `0007` applied
  live (with explicit authorization), then the golden-slice seeder re-run to upload clips and
  refresh rows. This is best practice for a content-heavy app — content updates without an App
  Store release.

## Testing & acceptance

**Automated gate (must stay green):** `npx tsc --noEmit && npx eslint . && npx jest`.
- New/updated unit tests: mapper carries `glide.audioUrl` + `pair.aAudioUrl/bAudioUrl`;
  `DrillScreen` say-it target == stimulus word; `PhraseSayIt` cue stage has no play orb and the
  compare stage exposes native/you playback.
- Snapshots updated intentionally for the centered `CardShell` and the reworked `PhraseSayIt`.

**Live phone walk (light + dark):**
1. Diphthong drill → "Meet the glide" plays an isolated glide (placeholder), **not** "lieta".
2. L/Ļ drill → say-it-back screen: the word shown == the word the play orb plays (see == hear).
3. Phrase say-it → English cue, no audio until after recording, then native-vs-you compare.
4. Phrase/word cards are **centered** and match the mockup composition.

**Acceptance:** all four above pass in both themes; the gate is green; the live golden-slice deck
is restored to a pristine state by the seeder.

## Out of scope (carried, not addressed here)

  re-seed, no code change.
- Adding tap-to-play on the drill choice buttons (data is made available; UX decision deferred).
- The broader content track (top-1000 native sign-off, full TTS corpus, ElevenLabs voice).
- Any other mockup screens not implicated by these three issues.
